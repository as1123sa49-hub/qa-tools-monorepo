# 測試案例自動產生工具

不需要安裝任何軟體，純前端網頁。上傳規格書 PDF（或匯入既有 TestCase）→ Gemini AI 分析 → 匯出 XLSX。

## 事前準備

需要一組 **Google Gemini API Key**，免費額度足夠日常使用。

1. 前往 [Google AI Studio](https://aistudio.google.com/app/apikey)
2. 點「Create API Key」
3. 複製 Key（格式：`AIzaSy...`）

## 使用方式

1. 在專案目錄啟動本機伺服器（**請勿**用 `file://` 直接開 `index.html`，否則匯出／CDN 可能異常）：
   ```powershell
   npx --yes serve .
   ```
   然後用瀏覽器開啟 `http://localhost:3000`
2. 貼上 Gemini API Key，點「💾 儲存」
3. 上傳規格書 PDF（支援拖曳）
4. 點「🚀 開始分析」
5. 等待 AI 完成（第 2 步有進度條與計時）
6. 產出後點「📊 匯出 XLSX」

## 分析模式

| 模式 | 說明 | 使用時機 |
|------|------|--------|
| 全量產出 | 針對整份規格書產出所有案例 | 首次分析、完整規格 |
| 差異比對 | 只針對新舊版本的變動部分產出案例 | 版本迭代更新 |
| 匯入Case比對新版 | 匯入既有測試案例（CSV/XLSX）再與新版規格比對 | 交接代操、多人協作 |
| 多規格生成 | 1 份主規格 PDF + 多份補充 PRD | 模組內容表 + 各區塊 PRD |
| 多批模組化 | 匯入 2～3 份 TestCase（CSV/XLSX），找相同規則並產出模組母版 | 分批產出後整理共用測項 |

差異比對需同時上傳**新版**與**舊版**規格書 PDF。

### 多規格生成

- **模組索引／分類（左側必填）**：如 `大平台 - 後台模組內容 玩家管理.pdf`
- **功能規格書（右側可多份）**：各功能完整 PRD（DOCX/PDF 等），**實際產 case 的內容來源**
- **分類邏輯**：優先依左側索引分類；對不上時退回規格書章節分類（同全量產出）
- **依規格書自動分批**（預設開啟）：≥ 2 份時，每批「索引 + 1 份規格書」
- 關閉自動分批時：維持一次合併全部 PRD 送 AI（大檔可能覆蓋不全）
- **提示詞**：右側獨立「多規格生成模式」分頁，與全量產出模板分開
- 合併後會標記「疑似重複」（同功能頁面 + 同規則族）；可勾選 AI 確認模糊群組
- 若有疑似重複，結果區會出現 **「🔍 重複比對」** 分頁：同群組案例相鄰排列，方便比對刪除；主表仍維持功能頁面/元件排序
- 可勾選列後「刪除選取」；分析進行中無法匯出 XLSX

### 多批模組化

- 批次 A、B 必填，C 選填；**不修改**原案例編號。
- 依「測試類型 + 規則族」分群（如 `NEG::EMAIL::REQUIRED`），可合併文案略有差異的相同驗證；入口跳轉與頁內編輯不併群。
- AB / BC / CA 相同 → 子模組；ABC 皆有 → 核心模組。
- 匯出 XLSX 含三個工作表：`模組母版`、`分類清單`、`摘要`。
- 可勾選「AI 確認疑似相同群組」（需 API Key）。

## Baseline 匯入格式（匯入Case比對新版）

匯入檔支援 `CSV` 與 `XLSX`，目前為單檔匯入。

必須包含下列欄位（標題需一致）：

- 狀態
- 取代者
- 編號
- 測試類型
- 類別
- 優先度
- 主模組
- 層級（L1 / L2 / L3）
- 功能頁面/元件
- 前置條件
- 測試標題
- 預期結果
- 規格來源
- 版本標籤

工具會以 `編號`（去前綴後）做去重比對：若出現重複，保留最新，舊筆標為失效（刪除線）。

## 快取功能（自動記憶上次版本）

每次分析成功後，工具會自動將新版規格書的文字內容儲存在瀏覽器 IndexedDB 中。

下次開啟時，舊版規格書區塊會出現提示欄：

```
💾 上次分析版本：V1_spec.pdf  儲存於 2026/3/24 14:32
   [📋 用作舊版比對]  [🗑 清除]
```

點「📋 用作舊版比對」即可直接以快取版本做差異比對，**不需重新上傳舊版 PDF**。

## 附加分析模式

當畫面上已有分析結果時，再次點「開始分析」會彈出確認：

- **「➕ 附加到現有結果」** — 新案例合併到現有表格
- **「🗑 清空並重新分析」** — 清除現有案例，重新產出

## 結果表格說明

| 欄位 | 說明 |
|------|------|
| 編號 | 類型縮寫_類別縮寫_序號（如 `POS_PLT_001`） |
| 測試類型 | 正面 / 負面 / 邊界 |
| 類別 | 平台 / 大廳 / 房內 / 異常處理 / 後台 |
| 優先度 | P0（核心）/ P1（一般）/ P2（UI/特效） |
| 主模組 | 通用功能、會員系統、金流系統、KYC審核、遊戲大廳 等 |
| 層級 | L1 / L2 / L3（測試案例多為 L2 或 L3） |
| 功能頁面/元件 | 該層節點名稱；差異比對含 `[新增]`/`[變更]`；AI 自創加 `[AI自創]` |
| 規格來源 | 頁碼或章節編號 |
| 版本標籤 | 分析日期（格式：`YYYYMMDD_v1`） |

預設排序：依規格書 `◆` 章節順序（主模組 → L2 區塊 → 功能頁面/元件），同一節點內再依測試類型與優先度。

> ⚠️ 表格中標紅色 `—` 的欄位表示 AI 未正常產出，需手動補齊

## 傳給其他人使用

把整個 `test-case-generator/` 資料夾壓縮成 zip 傳給對方，解壓後直接用瀏覽器開啟 `index.html` 即可。
對方需自備 Gemini API Key，API Key 僅儲存於本機 `localStorage`，不會外傳。
